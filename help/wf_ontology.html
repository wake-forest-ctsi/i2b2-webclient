Ontology Details

General Notes

  • The number appearing after a leaf is rough ballpark count of the number of instances("facts") of that concept in the data warehouse.  This is not a count of patients with those facts.  These numbers are updated periodically but not as frequently as new data is added to the warehouse.  These counts also help guide the i2b2 system in creating more efficient database queries on the backend.
  • The blue circle icon (e.g.: \WFUBMC\Diagnoses\Admitting Diagnosis) is called a "modifier", dragging a diagnosis over to a query group would query for all instances of that diagnosis; instead dragging over the Admitting Diagnosis modifier under that diagnosis, would only query for that diagnosis where it is in the EHR as the Admitting Diagnosis for an encounter.
  • There is a process for indentifying new concepts that are not currently in the ontology, but they may be missing for a while.  If there is a new medication, new department, etc. that you need to query, contact the data warehouse team and it can probably be added manually ahead of schedule.

    • Religion

    • Vital Status

      • Description: Y = Known deceased, N = Not known to be deceased
      • Dimensional Query: This is available per patient and not per visit.
      • Availability: If the patient died under our care, the value is most likely 'Y', if the patient died when not at the hospital, the value may be 'Y', but there is no assurance that we were ever notified of the death, so the value could be erroneously 'N'.

    • ZIP Code

      • Description: Patient's primary address zip code at the time of each visit.
      • Availability: 5-digit zip when available.  Some values are only the first 3-digits of the zip due to historical obfuscation to protect PHI.  The longer 5+4 zip is never recorded in the data warehouse, but is available when patient mailing addresses are requested.

  • Diagnoses

      • Description:  ICD-9 codeset.  Technically ICD-9 Procedures subset may live under here too, though they are also wholly contained in Procedures ontology.  ICD-10 available later in 2015.
      • Availability: 20+ years

  • Laboratory 

      • Description:  Laboratory tests and panels and results when available.
      • Availability:  20+ years.  Some sets are notably missing:  Blood bank is a separate system that has not been connected yet; Sendout results are notoriously not input fully or at all in some cases, occassionally they are partially entered into notes, but not into the usual discrete fields.
      • Query Set Value:  Lab result values can be tested with the Set Value functionality.  Result types, units, and ranges may differ for the same test depending on how long ago the sample was processed and by which laboratory.  When both numeric and textual results are available for a test, Set Value options are determined by which one is more prevalent overall.  Similarly when Set Value offers preset selectable values, the options have been determined by which are most prevalent overall.

    • LOINC By CPT

      • Description:  LOINC codes organized by the same structure as CPT codes.
      • Availability:  Mappings from WFU Codes were done by hand using the internally published Dept. of Pathology Laboratory Handbook to cover the top 80% of labs performed at the hospital as of 2011.  As such, available LOINC codes are a subset of WFU Codes, and therefore it is preferred to use WFU Codes for completeness.

    • LOINC By Number

      • Description:  LOINC codes navigable by number.  Items in this tree are identical to items in the LOINC By CPT tree, just organized differently.
      • Availability:  Mappings from WFU Codes were done by hand to cover the top 80% of labs performed at the hospital as of 2011.  As such, available LOINC codes are a subset of WFU Codes, and therefore it is preferred to use WFU Codes for completeness.

    • WFU Codes

      • Description:  Proprietary code set of lab codes, organized by Order/Panel (e.g.: CBCD) and by lab Test (e.g.: RBC)

  • Medication

      • Description:  RxNorm codes
      • Availability:  15+ years.  Mappings from WFU medications were initially done programmatically using the RxNav API.  The EPIC install included a custom mapping to RxNorm.  These two efforts have been merged.  Though in many cases a medication is only specified in the ontology by its ingredient, the specific drug may be available to be returned with a data request.  Also note we may have record of prescriptions ordered, but not of fills unless it was done at our own pharmacy.

    • Drug Products by Generic Ingredient Combinations

      • Description:  RxNorm codes organized alphabetically by ingredient.  Items in this tree are largely the same as what is available under the NDF-RT tree, though there are some concepts unique to this set.

    • NDF-RT

      • Description:  RxNorm codes organized by the NDF-RT structure.  Items in this tree are largely the same as what is available under the Drug Products tree, though there are some concepts unique to this set.

  • Procedures

      • Description:  CPT codeset.  These codes are mainly in the context of billing, and not infrequently hospital billed CPTs do not necessarily match hospital ordered CPTs.
      • Availability: 15+ years

  • Providers

      • Description:  Providers are organized by EPIC Department, and then obfuscated to a simple sequenced number.  If you need to identify specific providers, the BRSA Data Coordinators who handle data requests can map the EPIC Provider ID to the i2b2 obfuscated ID for you. 
      • Availability:  For every fact where available, so dependent on the data category in question.
      • Fact Queries:  Dragging over the provider number folder (e.g. \Providers\Allergy\00001\) finds all facts which are labeled with that provider.  Dragging the entire department folder works the same way, identifying facts associated with that department's providers.
      • Dimensional Queries:  Dragging over the specific leaves under that folder (e.g. \Providers\Allergy\00001\Admitting) finds all encounters for which that provider is named the Admitting Provider.  (Likewise for Discharging, Family/PCP, Referring provider leaves)

  • Reports

      • Description:  This is a "beta" set of radiology reports.  Querying will identify only if the report exists, but there is no functionality to query the text within that report.  Full notes and reports can be returned with BRSA Data Requests.
      • Availability:  2010-2011 only.

    • Department

      • Description:  Department associated with an encounter.  Typically admission department for Inpatient visits.
      • Availability:  All time, 30+ years.

    • Exists

      • Description:  This is a simple query to ask if any encounters exist.  This primarily is used if you want to find all patients seen within a date range not by any specific criteria, as date range functionality requires a contextual concept.
      • Availability:  All time.

    • Length of stay

      • Description:  Discharge date minus admit date, in most cases rounded.
      • Availability:  All time.

    • Office

      • Description:  This data point will probably go away and/or be replaced by an updated load.  It represented the physical clinic a patient was seen at, different from the department concept.
      • Availability:  15+ years until EPIC conversion.

    • Visit Type

      • Description:  Whether the encounter is classified as Inpatient or Outpatient.  Be forewarned this distinction is not always clearcut, as in such cases as patients under observation who are using beds.
      • Availability:  All time.

  • Vitals

    • Blood Pressure

      • Description:  Separate Systolic and Diastolic values.  Choose same financial encounter and Items Instance will be the same to query from the same measurement and apply Position or Vital Source modifiers.
      • Units:  mmHg
      • Availability: 15+ years, more frequently recorded after 2003.

    • BMI

      • Description:  Standard Body Mass Index calculation: 703 * ( kg / m2 ) / ( lb / in2 )
      • Units:  kg/m2
      • Availability: 15+ years, much more frequently recorded after EPIC conversion.

    • Height

      • Description:  Height of patient.
      • Units:  Inches.
      • Availability: 15+ years, much more frequently recorded after EPIC conversion.

    • Pulse

      • Description:  Pulse rate.
      • Units:  Beats per minute.
      • Availability: 15+ years, more frequently recorded after 2003.

    • Respiration Rate

      • Description:  Breathing rate.
      • Units:  Breaths per minute.
      • Availability: 15+ years, more frequently recorded after 2003.

    • Temperature

      • Description:  Body temperature.
      • Units:  Degrees Fahrenheit.
      • Availability: 15+ years, more frequently recorded after 2003.

    • Weight

      • Description:  Weight of patient.
      • Units:  Pounds (lbs).
      • Availability: 15+ years, much more frequently recorded after EPIC conversion.
